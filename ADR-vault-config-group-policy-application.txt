[ADR] Vault Group Policy Application Configuration Support in Terraform Vault Provider

Summary
This ADR introduces Terraform Vault Provider support for managing Vault's group policy application mode using the Terraform Plugin Framework. The implementation includes a singleton resource for managing the global configuration that controls how identity group policies are applied across namespaces in Vault Enterprise.

Background
The Vault group policy application feature in HashiCorp Vault Enterprise provides the ability to configure how policies attached to identity groups are applied across namespace boundaries. While Vault has supported this feature through the /sys/config/group-policy-application API endpoint since version 1.13.8, the Terraform Vault Provider has lacked a dedicated resource for managing this configuration.

Currently, users must rely on generic vault_generic_endpoint resources or manual vault CLI commands to configure group policy application mode, leading to:
* Lack of type safety and validation
* Complex state management for singleton configuration
* Poor developer experience with limited IDE autocomplete support
* Increased risk of configuration errors
* No enforcement of namespace restrictions (must be root/admin)
* Difficulty understanding the impact of configuration changes

The group policy application configuration is particularly important for:
* Multi-tenant environments: Controlling policy scope across namespace hierarchies
* Security compliance: Ensuring policies apply consistently or are properly isolated
* Enterprise deployments: Managing complex namespace structures with shared identity groups
* Access control: Determining when group policies take effect based on token namespace

Proposal
This ADR proposes adding Terraform Vault Provider (TFVP) support for Vault's group policy application configuration using the Terraform Plugin Framework. The implementation will introduce a singleton resource type that represents the global configuration for group policy application mode.

The proposed resource aligns with the Vault API endpoint and provides:
* Type-safe configuration management using Plugin Framework schema definitions
* Automated state tracking and drift detection
* Comprehensive validation of configuration parameters (only two valid modes)
* Enforcement of namespace restrictions (root or administrative namespace only)
* Import support for existing configurations
* Clear documentation of the impact of each mode
* Enterprise-only validation

Implementation
The implementation adds a new singleton resource class to the Terraform Vault Provider using the Terraform Plugin Framework. The component has clearly defined responsibilities and maps directly to the Vault API endpoint.

This change replaces the use of generic endpoint configurations with a dedicated, strongly typed Terraform resource that provides better validation, error handling, and state management for this critical global configuration.

Resource Architecture
The group policy application configuration implementation follows the established pattern used in the provider:

Configuration Resource
vault_config_group_policy_application: Manages the global group policy application mode configuration

Plugin Framework Implementation
The resource is implemented using the Terraform Plugin Framework, following the patterns established in recent provider resources like config_ui_header and password_policy.

Sample Schema Pattern:
func (r *ConfigGroupPolicyApplicationResource) Schema(ctx context.Context, req resource.SchemaRequest, resp *resource.SchemaResponse) {

Detailed Resource Specification
Resource: vault_config_group_policy_application

Manages the global group policy application mode for Vault Enterprise, controlling how policies attached to identity groups are applied across namespace boundaries.

* API Endpoint: POST/GET /sys/config/group-policy-application
* Vault Versions: 1.13.8+ (Enterprise only)
* Implementation: internal/vault/sys/config/group_policy_application.go

Important Notes
* This is a singleton resource - only one instance per Vault cluster
* Must be managed from root ("") or administrative ("admin") namespace
* Configuration is replicated between primary and secondary clusters
* Changing the mode affects all groups across all namespaces
* No true deletion - delete operation resets to default value
* Requires Vault Enterprise
* Resource ID is always "config" (constant)

Mode Descriptions:
* within_namespace_hierarchy (default): Policies only apply when the token authorizing a request was created in the same namespace as the group, or a descendent namespace. This is the historical behavior.
* any: Group policies apply to all members of a group, regardless of what namespace the request token came from. This relaxes the namespace restriction.

Example Usage
# Set group policy application to "any" mode
resource "vault_config_group_policy_application" "config" {
  group_policy_application_mode = "any"
}

# Or use default mode explicitly
resource "vault_config_group_policy_application" "config" {
  group_policy_application_mode = "within_namespace_hierarchy"
}

# Note: Only one instance of this resource should exist per Vault cluster

Import
terraform import vault_config_group_policy_application.config config

# Note: Import uses constant ID "config" as the identifier

Complete Usage Example
The following example demonstrates configuring group policy application mode with proper namespace handling:

# Configure group policy application mode
# This must be done in the root or administrative namespace
resource "vault_config_group_policy_application" "config" {
  # Set to "any" to allow group policies to apply regardless of token namespace
  group_policy_application_mode = "any"
  
  # Namespace must be root (empty string) or "admin"
  # namespace = ""  # Root namespace (default)
}

# Example: Switching back to default mode
resource "vault_config_group_policy_application" "config" {
  group_policy_application_mode = "within_namespace_hierarchy"
}

Implementation Considerations

Security
* Configuration affects all groups across all namespaces globally
* Changing from "within_namespace_hierarchy" to "any" relaxes security boundaries
* Should be carefully considered in multi-tenant environments
* Audit logs capture all configuration changes
* TLS should be enforced for production Vault connections
* Requires appropriate Vault policies for the sys/config/group-policy-application path

Plugin Framework Requirements
* Resource uses Plugin Framework schema types (StringAttribute)
* stringvalidator.OneOf() ensures only valid modes are accepted
* Singleton pattern with constant ID "config"
* Resources extend base.ResourceWithConfigure for consistent client management
* Resources extend base.WithImportByID for import support
* Namespace validation enforces root/admin requirement
* Version check ensures Vault 1.13.8+
* Enterprise check ensures Vault Enterprise

Namespace Restrictions
The Vault API requires this endpoint to be called from the root or administrative namespace only. This is enforced in the implementation:

// Validate namespace - must be root or administrative
namespace := data.Namespace.ValueString()
if namespace != "" && namespace != "admin" {
    resp.Diagnostics.AddError(
        "Invalid Namespace",
        fmt.Sprintf("Group policy application configuration must be managed from the root "+
            "or administrative namespace. Current namespace: %q. "+
            "Please use namespace = \"\" (root) or namespace = \"admin\".", namespace),
    )
    return
}

Singleton Pattern
Unlike resources that manage multiple named instances (like UI headers), this resource manages a single global configuration:

* ID is always "config" (hardcoded constant)
* Only one instance should exist per Vault cluster
* No name parameter needed
* Delete operation resets to default value instead of removing configuration
* Import always uses ID "config"

Backward Compatibility
This resource is a net-new addition to the Terraform Vault Provider and does not affect existing resources. Users currently managing this configuration via vault_generic_endpoint can migrate to this dedicated resource through standard Terraform state management practices using terraform import.

Migration Example:
# Before (using generic_endpoint)
resource "vault_generic_endpoint" "group_policy" {
  path = "sys/config/group-policy-application"
  data_json = jsonencode({
    group_policy_application_mode = "any"
  })
}

# After (using dedicated resource)
resource "vault_config_group_policy_application" "config" {
  group_policy_application_mode = "any"
}

# Migration steps:
# 1. Add new resource to configuration
# 2. Import existing config: terraform import vault_config_group_policy_application.config config
# 3. Remove old generic_endpoint resource
# 4. Run terraform plan to verify no changes

Conclusion
This ADR proposes a comprehensive implementation of group policy application configuration support in the Terraform Vault Provider using the Terraform Plugin Framework. The addition of a dedicated singleton resource will significantly improve the developer experience, provide type safety, enable better state management, and align with Terraform best practices.

The implementation covers all aspects of group policy application configuration including:
* Mode management (within_namespace_hierarchy vs any)
* Namespace restriction enforcement (root/admin only)
* Import support for existing configurations
* Version and Enterprise validation
* Clear error messages for common issues
* Comprehensive validation and error handling
* Singleton pattern for global configuration

The use of Plugin Framework aligns with HashiCorp's recommended approach for new Terraform provider development and ensures consistency with recent provider resources like config_ui_header and password_policy.

This provides users with a complete, production-ready toolset for managing Vault's group policy application mode via Terraform, enabling proper configuration of multi-tenant environments and namespace-based access control in Vault Enterprise.


SECURITY CONSIDERATIONS


Impact of Mode Changes:

within_namespace_hierarchy (Default):
* Policies only apply when token is from same or descendent namespace
* Maintains strict namespace isolation
* Recommended for multi-tenant environments
* Historical behavior - most restrictive

any:
* Policies apply regardless of token namespace
* Relaxes namespace boundaries
* Useful for shared identity groups across namespaces
* Should be carefully considered for security implications
