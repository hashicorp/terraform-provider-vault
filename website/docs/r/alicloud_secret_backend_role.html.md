# vault\_alicloud\_secret\_backend\_role

Creates a role on an AliCloud Secret Backend for Vault. Roles are used to map credentials to the policies or RAM roles that define their permissions.

The AliCloud secrets engine dynamically generates AliCloud access credentials based on RAM policies or by assuming RAM roles. This provides a secure way to grant temporary, limited-privilege credentials to applications and services.

~> **Important** All data provided in the resource configuration will be
written in cleartext to state and plan files generated by Terraform, and
will appear in the console output when Terraform runs. Protect these
artifacts accordingly. See
[the main provider documentation](../index.html)
for more details.

## Example Usage

### Role with RAM Role ARN (Recommended)

The recommended approach is to use an existing RAM role. Vault will generate temporary credentials by assuming this role.

```hcl
resource "vault_alicloud_secret_backend" "alicloud" {
  access_key    = "LTAI..."
  secret_key_wo = "your-secret-key"
}

resource "vault_alicloud_secret_backend_role" "role" {
  mount    = vault_alicloud_secret_backend.alicloud.path
  name     = "deploy-role"
  role_arn = "acs:ram::123456789012:role/MyApplicationRole"
  ttl      = 3600
  max_ttl  = 7200
}
```

### Role with Remote Policies

Use remote policies to reference existing system or custom RAM policies.

```hcl
resource "vault_alicloud_secret_backend_role" "role" {
  mount = vault_alicloud_secret_backend.alicloud.path
  name  = "oss-reader"
  
  # Reference multiple AliCloud RAM policies
  remote_policies = "name:AliyunOSSReadOnlyAccess,type:System,name:AliyunECSReadOnlyAccess,type:System"
  
  ttl     = 3600
  max_ttl = 7200
}
```

### Role with Inline Policies (Deprecated)

Inline policies allow you to define custom permissions directly. However, `remote_policies` is preferred for better maintainability.

```hcl
resource "vault_alicloud_secret_backend_role" "role" {
  mount = vault_alicloud_secret_backend.alicloud.path
  name  = "custom-permissions"
  
  inline_policies = jsonencode({
    Statement = [
      {
        Action = [
          "oss:GetObject",
          "oss:PutObject"
        ]
        Effect = "Allow"
        Resource = [
          "acs:oss:*:*:my-bucket/*"
        ]
      }
    ]
    Version = "1"
  })
  
  ttl     = 3600
  max_ttl = 7200
}
```

## Argument Reference

The following arguments are supported:

* `namespace` - (Optional) The namespace to provision the resource in.
  The value should not contain leading or trailing forward slashes.
  The `namespace` is always relative to the provider's configured [namespace](/docs/providers/vault/index.html#namespace).
  *Available only for Vault Enterprise*.

* `mount` - (Required) The path where the AliCloud secret backend is mounted,
  with no leading or trailing `/`s.

* `name` - (Required) The name of the role. Must be unique within the backend.

* `role_arn` - (Optional) The ARN of the RAM role to assume. When specified, Vault will generate temporary credentials by assuming this role. The role's trust policy must allow the access key configured in the backend to assume it. Cannot be used with `inline_policies` or `remote_policies`. **This is the recommended approach for production use.**

* `inline_policies` - (Optional) A JSON string containing an inline RAM policy document. The policy should be a valid AliCloud RAM policy. Cannot be used with `role_arn`. **Deprecated: use `remote_policies` instead for better maintainability.**

* `remote_policies` - (Optional) A comma-separated string of remote policy names and types in the format: `name:PolicyName,type:PolicyType,name:PolicyName2,type:PolicyType2`. The `type` must be either `System` (for AliCloud managed policies) or `Custom` (for customer-managed policies). Cannot be used with `role_arn`.

* `ttl` - (Optional) The default TTL for credentials issued by this role, in seconds. Defaults to the system/backend default TTL.

* `max_ttl` - (Optional) The maximum TTL for credentials issued by this role, in seconds. Defaults to the system/backend maximum TTL.

~> **Note:** You must specify exactly one of `role_arn`, `inline_policies`, or `remote_policies`. The role will fail validation if none is provided. Also vault accepts combination of both `inline_policies` and `remote_policies`.

In addition to the arguments above, the following attributes are exported:

## Import

AliCloud secret backend roles can be imported using the full path to the role
in the format: `<mount>/role/<name>`, e.g.

```shell
$ terraform import vault_alicloud_secret_backend_role.role alicloud/role/deploy-role
```

~> **Note:** After import, the `mount` field will be set to the mount path portion of the imported ID.

## Credential Types Comparison

| Method | Use Case | Pros | Cons |
|--------|----------|------|------|
| `role_arn` | Production deployments | Most secure, leverages existing RAM roles, follows Alicloud best practices | Requires pre-existing RAM role |
| `remote_policies` | Dynamic permissions with existing policies | Reuses existing policies, easier to manage | Creates new RAM users (less secure than assuming roles) |
| `inline_policies` | Quick testing, custom permissions | Flexible, no external dependencies | Deprecated, harder to maintain, creates RAM users |
