# vault\_os\_secret\_backend\_account

Manages account configurations in the OS Secrets Engine. Accounts represent operating system user
accounts on remote hosts that Vault will manage, including automatic password rotation.
This resource requires Vault 2.0.0 or later.

~> **Important** The `password` field is write-only and will not be read back from Vault.
All data provided in the resource configuration will be written in cleartext to state and plan
files generated by Terraform. Protect these artifacts accordingly. See
[the main provider documentation](../index.html) for more details.

See the [Vault documentation](https://www.vaultproject.io/docs/secrets/os) for more information.

## Example Usage

### Basic Account Configuration

```hcl
resource "vault_os_secret_backend" "os" {
  path = "os"
}

resource "vault_os_secret_backend_host" "server" {
  mount   = vault_os_secret_backend.os.path
  name    = "web-server"
  type    = "ssh"
  address = "192.168.1.100"
  port    = 22
}

resource "vault_os_secret_backend_account" "admin" {
  mount    = vault_os_secret_backend.os.path
  host     = vault_os_secret_backend_host.server.name
  name     = "admin-account"
  username = "admin"
  password = "initial-secure-password-123"
}
```

### Account with Rotation Configuration

```hcl
resource "vault_os_secret_backend" "os" {
  path = "os"
}

resource "vault_os_secret_backend_host" "database" {
  mount   = vault_os_secret_backend.os.path
  name    = "db-server"
  type    = "ssh"
  address = "10.0.1.50"
  port    = 22
}

resource "vault_os_secret_backend_account" "dbadmin" {
  mount           = vault_os_secret_backend.os.path
  host            = vault_os_secret_backend_host.database.name
  name            = "dbadmin"
  username        = "postgres"
  password        = "initial-password-456"
  rotation_period = 86400  # Rotate every 24 hours
  rotation_window = 3600   # Allow 1 hour window for rotation
}
```

### Account with Scheduled Rotation

```hcl
resource "vault_os_secret_backend" "os" {
  path = "os"
}

resource "vault_os_secret_backend_host" "app_server" {
  mount   = vault_os_secret_backend.os.path
  name    = "app-01"
  type    = "ssh"
  address = "192.168.1.200"
  port    = 22
}

resource "vault_os_secret_backend_account" "service" {
  mount             = vault_os_secret_backend.os.path
  host              = vault_os_secret_backend_host.app_server.name
  name              = "service-account"
  username          = "appuser"
  password          = var.initial_password
  rotation_schedule = "0 3 * * 0"  # Rotate every Sunday at 3 AM
}
```

## Argument Reference

The following arguments are supported:

* `namespace` - (Optional) The namespace to provision the resource in.
  The value should not contain leading or trailing forward slashes.
  The `namespace` is always relative to the provider's configured [namespace](/docs/providers/vault/index.html#namespace).
   *Available only for Vault Enterprise*.

* `mount` - (Required) The path where the OS secrets engine is mounted.

* `host` - (Required) The name of the host where this account exists.

* `name` - (Required) Unique name for the account within the host.

* `username` - (Required) The username of the operating system account on the remote host.

* `password` - (Required, Sensitive) The initial password for the account. **This field is write-only and will not be read back from Vault.** The password will be stored in Terraform state but never retrieved from the Vault API.

* `rotation_period` - (Optional) The period in seconds between automatic password rotations. If not set, the password will not be automatically rotated.

* `rotation_window` - (Optional) The window in seconds during which rotation can occur. Must be less than `rotation_period`. If not set, rotation can occur at any time during the period.

* `rotation_schedule` - (Optional) A cron-style schedule for password rotation (e.g., `"0 3 * * 0"` for weekly on Sunday at 3 AM). If set, this takes precedence over `rotation_period`.

## Attributes Reference

In addition to the arguments above, the following attributes are exported:

* `id` - The ID of the account in the format `<mount>/host/<host>/account/<name>`.

* `last_vault_rotation` - (Computed) The timestamp of the last password rotation performed by Vault.

* `next_vault_rotation` - (Computed) The timestamp when the next password rotation is scheduled to occur.

## Import

OS Secret backend account can be imported using the format `<mount>/host/<host>/account/<name>`, e.g.

```
$ terraform import vault_os_secret_backend_account.admin os/host/web-server/account/admin-account
```

~> **Note:** When importing, the `password` field will not be populated since it is write-only.
You will need to provide the password in your configuration or use `lifecycle { ignore_changes = [password] }`
to prevent Terraform from trying to update it.

## Notes

* This resource requires Vault 2.0.0 or later.
* The account must reference an existing host configuration.
* The `password` field is write-only for security reasons. Once set, Vault will manage the password through rotation, but the current password cannot be retrieved through the API.
* When Vault rotates the password, the `last_vault_rotation` and `next_vault_rotation` attributes will be updated automatically.
* The `rotation_schedule` uses standard cron syntax. If both `rotation_schedule` and `rotation_period` are set, `rotation_schedule` takes precedence.
* The `rotation_window` defines a time window during which rotation can occur, helping to avoid rotating passwords during peak usage times or maintenance windows.
* After the initial password is set, Vault will rotate it according to the configured schedule. Applications should retrieve the current password from Vault rather than storing it.
* Changing `mount`, `host`, or `name` will cause the resource to be recreated.
* The computed fields `last_vault_rotation` and `next_vault_rotation` are automatically updated by Vault and reflect the rotation status.

## Security Considerations

* Store the initial `password` in a secure location such as Terraform variables or a secrets management system.
* Use `sensitive = true` in variable definitions for passwords.
* Consider using `lifecycle { ignore_changes = [password] }` after initial creation to prevent accidental password updates through Terraform.
* Ensure Terraform state files are properly secured as they will contain the initial password in cleartext.
* Applications should retrieve credentials from Vault dynamically rather than storing them.
